## v2.0 更新内容

**项目作用域模式**（最大变化！）—— Law Zero 现在默认只扫描目标项目内的文件。全局文件（CLAUDE.md、MEMORY.md、lessons）变为只读，除非你明确要求全局扫描。防止多项目共用工具时的交叉污染。

| 特性 | v1.0 | v2.0 |
|------|------|------|
| 一致性扫描 | 始终全局 | **默认项目作用域** |
| 全局文件 | 自由修改 | **项目模式下只读** |
| 跨项目安全 | 未处理 | **项目间完全隔离** |
| 扫描模式 | 只有一种 | **两种：项目模式（默认）+ 全局模式** |

## 问题

你花了几个小时教 Claude Code 你项目的规则、工具和流程。然后你升级了一个脚本、添加了新技能、改了配置——突然：

- 旧指令和新指令打架
- 已废弃的方法还在被推荐
- 同样的错误再次发生，因为教训丢了
- `CLAUDE.md` 说一套，`SKILL.md` 说另一套
- **更糟：修复项目A的文档意外破坏了项目B**（v2.0 解决了这个问题！）

## 解决方案：Law Zero（第零法则）

Law Zero 是 Claude Code 的**自我进化框架**。它是一条凌驾于所有规则之上的"第零法则"，确保 AI 助手在升级后自动维护一致性。

任何东西升级时，Law Zero 触发 4 步协议：

```
升级 → 删除旧版 → 一致性扫描 → 验证汇报
```

没有复杂配置，不需要外部工具。放到 `CLAUDE.md` 顶部就行。

## 两种扫描模式（v2.0 核心）

| 模式 | 触发方式 | 扫描范围 | 全局文件 |
|------|---------|---------|---------|
| **项目模式**（默认） | "对XX项目做第零法则" | 仅该项目目录下的文件 | **只读不写** |
| **全局模式** | "做全局第零法则" | 所有项目+全局文件 | 可读可写 |

### 为什么项目模式是默认？

- 项目A和项目B可能共用同一个工具（如 Playwright），但使用方式不同
- 在项目A的扫描中改了全局文件 → 可能破坏项目B的行为
- 多轮累积 → 全局文件被来回修改 → 越来越混乱
- **每个项目的第零法则只优化自己，不影响其他项目**

### v2.0 诞生的真实故事

我们对小红书（XHS）自动回复项目做了第零法则扫描。扫描发现矛盾并"修复"了它们——包括修改了 `CLAUDE.md`、`lessons/web.json` 和 `MEMORY.md` 这些全局文件。

这些修改对XHS是正确的，但影响了微信自动化项目（共用 Page Agent 工具）。下一次微信发文件操作就发到了错误的人——因为全局配置被XHS扫描改了。

**v2.0 的解决方案**：默认项目模式。XHS扫描只修改XHS自己的文件（`SKILL.md`、`scripts/*`），全局文件只读参照。

## 快速开始

把以下内容加到你的 `CLAUDE.md` **最顶部**：

```markdown
## 第零法则：自我进化机制（最高优先级！所有规则之上！）

### 作用域规则（最重要！）

两种模式，默认项目模式：

| 模式 | 触发方式 | 扫描范围 | 全局文件 |
|------|---------|---------|---------|
| **项目模式**（默认） | "对XX项目做第零法则" | 仅项目目录 | 只读不写 |
| **全局模式** | "做全局第零法则" | 所有文件 | 可读可写 |

项目模式铁律：全局文件（CLAUDE.md、MEMORY.md、lessons）只读不写。

### 执行流程（4步，缺一不可）

1. **升级实施** — 完成新版本部署
2. **删除旧版** — 彻底清除作用域内所有被替代的旧内容
3. **一致性扫描** — 扫描作用域内所有文件，发现矛盾立即在范围内修正
4. **验证汇报** — 列出完整的文件修改清单，确认一致性
```

详细模板见 [`templates/CLAUDE.md`](templates/CLAUDE.md)。

## 4步自我进化协议

**第1步 — 升级实施**
- 完成新版本的部署/修改/替换

**第2步 — 删除旧版**
- 彻底删除作用域内所有被替代的旧内容
- 绝不允许新旧并存
- 项目模式：只清理项目目录下的文件
- 全局模式：可清理所有文件

**第3步 — 一致性扫描**
- **项目模式**：扫描项目的SKILL.md、scripts/*、data/*。读（不写）全局文件检查矛盾
- **全局模式**：扫描所有CLAUDE.md、memory、lessons、SKILL.md文件
- 发现矛盾 → 在作用域内修正，不越界

**第4步 — 验证汇报**
- 确认作用域内所有文件一致
- **必须列出修改的文件清单**，让用户清楚改了什么

## 教训学习系统

动态的、基于文件的知识系统，捕获教训、防止重蹈覆辙。

```
~/.claude/lessons/
  ├── general.json      # 通用原则
  ├── web.json          # 网页操作教训
  ├── wechat.json       # 微信操作教训
  ├── git.json          # Git操作教训
  └── <任意主题>.json    # 按需自动创建
```

**三条铁律：**
1. **操作前查教训** — 扫描相关教训文件，检查已知陷阱
2. **犯错后写教训** — 立刻写入文件（"说记住了不算数，写入文件才算数"）
3. **分类自动扩展** — 没有预设限制，需要新分类直接创建新文件

## 设计哲学

**1. 项目隔离（v2.0）** — 每个项目只优化自己，不碰其他项目

**2. 约定优于配置** — 不需要外部工具，通过自然语言指令工作

**3. 单一数据源** — 每份知识只存一个地方，其他地方指向源

**4. 安全优先** — 验证步骤防止误删，汇报步骤提供审计日志

**5. 渐进增强** — 从30秒的最小配置开始，按需扩展

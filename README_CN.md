## v3.0 更新内容

**安全升级 + 可回滚**（最大变化！）—— Law Zero 不再立即删除旧版本。废弃文件移到 `.deprecated/` 目录并标注日期，经过 24-48 小时观察期确认新版稳定后才真正清理。这解决了级联故障的头号原因：新版还没经过生产验证就把旧版删了。

| 特性 | v1.0 | v2.0 | v3.0 |
|------|------|------|------|
| 一致性扫描 | 始终全局 | 默认项目作用域 | 默认项目作用域 |
| 全局文件 | 自由修改 | 项目模式下只读 | 项目模式下只读 |
| 跨项目安全 | 未处理 | 项目间完全隔离 | 项目间完全隔离 |
| 旧版处理 | 立即删除 | 立即删除 | **废弃→观察→清理** |
| 回滚能力 | 无 | 无 | **从 .deprecated/ 快速回滚** |
| 协议步骤 | 4步 | 4步 | **5步** |

## 问题

你花了几个小时教 Claude Code 你项目的规则、工具和流程。然后你升级了一个脚本、添加了新技能、改了配置——突然：

- 旧指令和新指令打架
- 已废弃的方法还在被推荐
- 同样的错误再次发生，因为教训丢了
- `CLAUDE.md` 说一套，`SKILL.md` 说另一套
- **更糟：修复项目A的文档意外破坏了项目B**（v2.0 解决了！）
- **再糟：旧版删了，新版有bug，无法回滚**（v3.0 解决了！）

## 解决方案：Law Zero（第零法则）

Law Zero 是 Claude Code 的**自我进化框架**。它是一条凌驾于所有规则之上的"第零法则"，确保 AI 助手在升级后自动维护一致性。

任何东西升级时，Law Zero 触发 5 步协议：

```
升级 → 废弃旧版 → 一致性扫描 → 观察期 → 清理
```

没有复杂配置，不需要外部工具。放到 `CLAUDE.md` 顶部就行。

---

## 两种扫描模式（v2.0 引入）

| 模式 | 触发方式 | 扫描范围 | 全局文件 |
|------|---------|---------|---------|
| **项目模式**（默认） | "对XX项目做第零法则" | 仅该项目目录下的文件 | **只读不写** |
| **全局模式** | "做全局第零法则" | 所有项目+全局文件 | 可读可写 |

### 为什么项目模式是默认？

- 项目A和项目B可能共用同一个工具（如 Playwright），但使用方式不同
- 在项目A的扫描中改了全局文件 → 可能破坏项目B的行为
- 多轮累积 → 全局文件被来回修改 → 越来越混乱
- **每个项目的第零法则只优化自己，不影响其他项目**

---

## 5步安全进化协议（v3.0 核心）

**第1步 — 升级实施**
- 完成新版本的部署/修改/替换

**第2步 — 废弃旧版（不删除！移到 .deprecated/）**
- 旧文件**不直接删除**，移到作用域内的 `.deprecated/` 目录
  - 项目模式：`项目目录/.deprecated/YYYY-MM-DD/`
  - 全局模式：`~/.deprecated/YYYY-MM-DD/`
- 移动时保留目录结构，方便回滚
- 在文档/代码中标注"已废弃，由XX替代"
- **为什么不直接删**：新版未经生产检验，旧版是唯一的安全网

**第3步 — 一致性扫描**
- **项目模式**：扫描项目的 SKILL.md、scripts/*、data/*。读（不写）全局文件检查矛盾
- **全局模式**：扫描所有 CLAUDE.md、memory、lessons、SKILL.md 文件
- 发现矛盾 → 在作用域内修正，不越界

**第4步 — 验证 & 观察期**
- 确认作用域内所有文件一致
- **必须列出修改的文件清单**，让用户清楚改了什么
- **观察期**：新版需在生产环境正常运行 24-48 小时
  - 观察期内出问题 → 从 `.deprecated/` 快速回滚
  - 观察期内不要清理 `.deprecated/`

**第5步 — 清理废弃文件（观察期结束后）**
- 新版稳定运行 24-48 小时后，确认无其他脚本/项目还在调用旧版
- 此时才真正删除 `.deprecated/` 中的旧文件
- 用户明确说"直接删不用观察"→ 可跳过观察期

## 为什么需要 v3.0？真实的痛

v2.0 解决了跨项目污染，但我们不断遇到另一个问题：

```
第1天：bridge.sh v7 → v8，立即删除 v7
第2天：v8 有竞态条件 bug（锁文件冲突）
第3天：无法回滚——v7 已经没了。只能在压力下现场 debug v8。
```

```
第1天：Chrome自动化改用默认配置文件，删除旧的 --user-data-dir 方案
第2天：Chrome 145 安全限制完全阻止新方案
第3天：必须凭记忆重建旧方案——浪费2小时。
```

用了 v3.0 之后：
```
第1天：bridge.sh v7 → v8，v7 移到 .deprecated/2026-02-20/
第2天：v8 有 bug → 从 .deprecated/ 拷贝回 v7，10秒恢复
第3天：从容修复 v8，重新部署，继续观察
第4天：v8 稳定 → 删除 .deprecated/
```

## 快速开始

把以下内容加到你的 `CLAUDE.md` **最顶部**：

```markdown
## 第零法则：自我进化机制（最高优先级！所有规则之上！）

### 作用域规则（最重要！）

两种模式，默认项目模式：

| 模式 | 触发方式 | 扫描范围 | 全局文件 |
|------|---------|---------|---------|
| **项目模式**（默认） | "对XX项目做第零法则" | 仅项目目录 | 只读不写 |
| **全局模式** | "做全局第零法则" | 所有文件 | 可读可写 |

### 执行流程（5步，缺一不可）

1. **升级实施** — 完成新版本部署
2. **废弃旧版** — 移到 `.deprecated/YYYY-MM-DD/`（不删除！）
3. **一致性扫描** — 扫描作用域内所有文件，发现矛盾立即修正
4. **验证 & 观察期** — 列出文件清单，新版运行 24-48 小时
5. **清理废弃文件** — 观察期通过后才真正删除 `.deprecated/`
```

详细模板见 [`templates/CLAUDE.md`](templates/CLAUDE.md)。

## 教训学习系统

动态的、基于文件的知识系统，捕获教训、防止重蹈覆辙。

```
~/.claude/lessons/
  ├── general.json      # 通用原则
  ├── web.json          # 网页操作教训
  ├── wechat.json       # 微信操作教训
  ├── git.json          # Git操作教训
  └── <任意主题>.json    # 按需自动创建
```

**三条铁律：**
1. **操作前查教训** — 扫描相关教训文件，检查已知陷阱
2. **犯错后写教训** — 立刻写入文件（"说记住了不算数，写入文件才算数"）
3. **分类自动扩展** — 没有预设限制，需要新分类直接创建新文件

## 设计哲学

**1. 项目隔离（v2.0）** — 每个项目只优化自己，不碰其他项目

**2. 安全升级（v3.0）** — 废弃优先于删除，始终保留回滚路径

**3. 约定优于配置** — 不需要外部工具，通过自然语言指令工作

**4. 单一数据源** — 每份知识只存一个地方，其他地方指向源

**5. 安全优先** — 验证步骤防止误删，汇报步骤提供审计日志

**6. 渐进增强** — 从30秒的最小配置开始，按需扩展
